What Are Calling Conventions?

Calling conventions are the rules programs follow when functions call each other. They decide:

  • How the CPU gets back to where it was (return address)
  • How local variables are preserved (callee-saved registers)
  • How parameters are passed (registers in 64-bit, stack in 32-bit)
  • How the stack frame is cleaned up (epilogue instructions)
  • What changes between 32-bit and 64-bit systems

How This Relates To Calling Conventions

Return Address

When you call a function, the CPU automatically pushes the current instruction pointer (RIP/EIP) onto the stack. On Return, the address is popped back into RIP which is how the CPU knows where to resume.

Local Variables & Callee-Saved Registers

Locals live below RBP in the frame (negative offsets). Callee-saved registers such as RBX, RBP, R12–R15 must be restored by the callee before returning. The caller trusts them to be unchanged. Caller-saved registers such as RAX, RDI, RSI, RDX, etc. may be overwritten by any call.

Parameter Passing

64-bit: first 6 integer args go in RDI, RSI, RDX, RCX, R8, R9 with no stack push needed. 32-bit (cdecl): all args are pushed right-to-left, accessed at positive offsets from EBP inside the callee.

Stack Cleanup (Epilogue)

The epilogue reverses the prologue: mov rsp, rbp collapses locals, pop rbp restores the caller's base pointer, ret pops the return address. In 32-bit cdecl, the caller also runs add esp, N to remove pushed args.

32-bit vs 64-bit

64-bit has more registers, uses them for parameters, which is faster and avoids memory accesses, has 8-byte pointers, and requires 16-byte stack alignment before a call. 32-bit relies on the stack for everything and has only 4-byte pointers, so a maximum of 4 GB is addressable.

Why It Matters for Security

The predictable stack layout that makes calling conventions work also makes stack buffer overflows dangerous. An attacker who overflows a local buffer can overwrite the return address then ret jumps to attacker-controlled code.
